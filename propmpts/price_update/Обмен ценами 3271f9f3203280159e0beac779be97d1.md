# Обмен ценами

Реализовать отдельный воркер для обмена с внешней системой по образцу обмена с системами 5post и Magnit, изучи код и детали реализации их и сделай так же. 

Цель данного воркера - обмен (загрузка) цен из внешней системы. Система предоставляет файл в формате .xml с актулальными ценами, их надо загружать в систему, вести журналы итд итп. 

```html
загружать цены с ftp:91.221.103.141 login:coffeeftp password:S4QKI0v1g0
```

файлы формата last_import_data_2026_03_18_13_30.xml где в названии файла есть год, месяц день и время: last_import_data_год_месяц_день_час_мин.xml

Загружать самый свежий файл с ftp раз в N минут

Интересующие нас цены хранятся в 

```xml
<import>
	<goods>
		<good>
			…
		</good>
	</goods>
</import>
```

- остальные разделы xml файла не рассматриваем.

пример товара:

```xml
<good id="412d0eea-df19-11eb-b9d2-005056c00008" parent="6b3e084d-d7bd-11ea-903c-005056c00008">
		<name>Roasted coffee beans VKUS ESPRESSO, 1 kg</name>
		<barcode/>
		<article>701</article>
		<count>2314</count>
		<prices>
			<trade currency="643">3828</trade>
			<base currency="643"/>
			<sale currency="643">3329</sale>
			<cost currency="643">1248,71</cost>
		</prices>
	</good>
```

тут trade, base, sale, cost - это различные виды цен. В будущем могут появиться дополнительные виду цен. trade - обычная цена, base- базовая, sale - акция или распродажа. Если задана цена sale (не пустая) это сигнал для нашей системы, что ***СЕЙЧАС*** товар продается со скидкой, и это сигнал для фронта что карточку товара надо как то особенно отображать(бейдж “распродажа”, “скидка” итп). 

Если при обмене у товара вид цен sale не задан, это значит что у нас в базе старая цена sale должна быть удалена. Тоже самое и для других видов цен. 

 cost - это себестоимость товара, по сути еще один вид цен. 

article - это артикул товара и полностью совпадает с нашим products.sku

count - остаток товара на складе магазина, его пока не надо подгружать, остатки товара будет вести отдельно внутри магазтина.

barcode - игнорировать.

Предлагаю хранить цены по видам, привязанные к public.products.

Вариант идеи предполагаемой структуры таблиц, где products - уже существующая таблица в нашей базе. 

```html
CREATE TABLE IF NOT EXISTS "products" (
	"id" serial NOT NULL UNIQUE,
	"sku" varchar(50) NOT NULL,
	PRIMARY KEY ("id")
);

CREATE TABLE IF NOT EXISTS "price_type" (
	"id" serial NOT NULL UNIQUE,
	"price_code" varchar(50) NOT NULL,
	"laber" varchar(255) NOT NULL, 
	PRIMARY KEY ("id")
);

CREATE TABLE IF NOT EXISTS "product_price" (
	"product_id" serial NOT NULL UNIQUE,
	"price_type_id" varchar(36) NOT NULL,
	"price" bigint NOT NULL
);

ALTER TABLE "product_price" ADD CONSTRAINT "product_price_fk0" FOREIGN KEY ("product_id") REFERENCES "products"("id");
ALTER TABLE "product_price" ADD CONSTRAINT "product_price_fk1" FOREIGN KEY ("price_type_id") REFERENCES "price_type"("id");
```

таблица price_type хранит типы цен: trade, base, sale, cost - по записи в таблице на каждый вид.

Вести журналы истории загрузки цен. Журнал показывает какие товары в базе были обновлены а какие - нет. Журналы вести посессионно.
