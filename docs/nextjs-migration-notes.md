# Миграция на Next.js — заметки

Дата: 2026-03-20

## Текущий стек фронтенда

- Vite + React + React Router (HashRouter, SPA)
- Tailwind CSS
- motion/react (framer-motion) для анимаций
- Yandex Maps (v3 API)
- Деплой: статика на nginx (dev.vkus.online)

## Оценка трудоёмкости миграции

### Что просто (1-2 дня)

- Tailwind CSS — работает без изменений
- Компоненты UI (.tsx) — переносятся как есть с добавлением `"use client"`
- API-вызовы (api.ts) — работают без изменений на клиенте

### Что потребует переработки (3-5 дней)

- Роутинг: HashRouter (`/#/catalog`) -> файловая структура (`app/catalog/page.tsx`)
- Все `useNavigate`, `Link`, `useParams` -> переписать на `next/navigation`
- Контексты (Cart, Auth, Favorites) -> обернуть в `"use client"` провайдеры
- Динамические маршруты: `/#/orders/:token` -> `app/orders/[token]/page.tsx`
- Яндекс Карты: загрузка скриптов через `next/script`

### Что создаст сложности (2-3 дня)

- Деплой: нужен Node.js сервер (PM2/Docker) вместо статики на nginx
  - Альтернатива: `output: "export"` для статики, но теряется SSR
- YooKassa redirect-flow с return_url — проверить совместимость
- motion/react — нужно `"use client"` на каждом компоненте с анимацией
- SEO-оптимизация каталога — переписать загрузку данных на серверные компоненты

## Итого

| Аспект              | Оценка                       |
|---------------------|------------------------------|
| Трудоёмкость        | ~2 недели работы             |
| Риск сломать что-то | Средний                      |
| Выгода для SEO      | Высокая (каталог, товары)    |
| Выгода для скорости | Умеренная                    |

## Плюсы Next.js по сравнению с текущей архитектурой

### Что улучшится

**1. SEO каталога.**
Сейчас поисковики видят пустую страницу — весь контент рендерится в браузере через JS. С Next.js Google/Yandex сразу получают готовый HTML с товарами, описаниями, ценами. Для интернет-магазина это критично.

**2. Скорость первой загрузки.**
Сейчас браузер скачивает ~1MB JS-бандл, только потом рисует страницу. Next.js отдаёт готовый HTML — пользователь видит контент мгновенно, JS догружается в фоне.

**3. Open Graph / соцсети.**
Ссылка на товар в Telegram/WhatsApp сейчас показывает пустую превью. С SSR — красивая карточка с названием, ценой, картинкой.

**4. Нормальные URL.**
Сейчас `dev.vkus.online/#/catalog` (хеш-роутинг). С Next.js — `vkus.online/catalog`, `vkus.online/product/espresso`. Чище, лучше для SEO, можно шарить.

**5. Инкрементальная регенерация (ISR).**
Каталог можно закешировать на сервере и обновлять раз в 5 минут — страницы будут отдаваться мгновенно без нагрузки на API при каждом запросе.

**6. API Routes.**
Можно перенести часть бэкенд-логики (webhook-и YooKassa, проксирование) прямо во фронт-проект, упростив архитектуру.

### Что НЕ улучшится

- Скорость работы после загрузки (SPA уже быстрый)
- Checkout / корзина / ЛК — это клиентская логика, SSR там не нужен
- Карты Yandex — всё равно клиентский рендеринг

### Вывод

Основной аргумент — SEO и шаринг ссылок. Если магазин планирует привлекать трафик из поиска, Next.js окупится. Если трафик приходит из рекламы/соцсетей напрямую — текущий стек достаточен.

## Альтернативы Next.js для решения SEO на текущей архитектуре

### 1. Prerender (самый популярный)

Отдельный сервис рендерит страницы в headless-браузере и отдаёт готовый HTML поисковикам.

Как работает:
- Nginx проверяет User-Agent запроса
- Если бот (Googlebot, YandexBot, Telegram) — проксирует на prerender-сервис
- Если обычный пользователь — отдаёт SPA как обычно

Варианты:
- **prerender.io** — SaaS, платный (~$9/мес за 250 страниц). Ноль настройки
- **prerender-node** — self-hosted, бесплатный. Puppeteer рендерит страницы и кеширует
- **rendertron** (Google) — аналог от Google, open-source

Плюсы: не трогаешь фронтенд вообще, работает с любым SPA.
Минусы: задержка рендеринга для ботов (2-5 сек), нужен сервер с Chrome/Puppeteer.

### 2. react-snap (статическая пре-генерация)

На этапе сборки (`npm run build`) запускает headless-браузер, проходит по всем маршрутам, сохраняет HTML.

Плюсы: никакого runtime-сервиса, просто статика.
Минусы: не работает с HashRouter (`/#/`), нужно сначала перейти на BrowserRouter. Ломается если маршруты динамические (каждый товар).

### 3. SSG через Vite-plugin-ssr / vike

Плагин для Vite, добавляющий SSR/SSG без переезда на Next.js.

Плюсы: остаёшься на Vite, минимальные изменения.
Минусы: менее зрелый чем Next.js, меньше документации.

### 4. Meta-теги через SPA (минимальный подход)

Для Open Graph (Telegram/WhatsApp превью) — динамически генерировать `<meta>` теги на бэкенде:
- Nginx перехватывает запрос к `/product/espresso`
- Бэкенд отдаёт минимальный HTML с правильными `og:title`, `og:image`, `og:description`
- Скрипт SPA загружается и берёт управление

Плюсы: решает проблему шаринга ссылок без prerender.
Минусы: не решает SEO полностью (тело страницы пустое для ботов).

### Сравнение подходов

| Подход          | Сложность | Решает SEO | Решает OG | Нужен сервер |
|-----------------|-----------|------------|-----------|--------------|
| prerender.io    | Низкая    | Да         | Да        | Нет (SaaS)   |
| Self-hosted     | Средняя   | Да         | Да        | Да (Puppeteer)|
| react-snap      | Средняя   | Частично   | Да        | Нет          |
| Meta-теги       | Низкая    | Нет        | Да        | Нет          |
| Next.js         | Высокая   | Да         | Да        | Да (Node.js) |

Самый быстрый старт — prerender.io (подключается за час) или meta-теги на бэкенде если нужны только превью в мессенджерах.

## Рекомендация

Сейчас не стоит мигрировать на Next.js — активная фаза разработки функциональности. Миграция лучше когда функционал стабилизируется.

Если SEO нужно срочно — проще добавить prerender для каталога (prerender.io или self-hosted) без переезда на Next.js.

## Self-hosted Prerender — подробности

### Схема работы

```
Посетитель → Nginx → отдаёт SPA (index.html + JS) → рендерится в браузере

Googlebot  → Nginx → видит бота по User-Agent → проксирует на Prerender-сервис
                                                        ↓
                                                  Puppeteer открывает URL
                                                  Ждёт пока JS отрисует
                                                  Возвращает готовый HTML
                                                        ↓
                                                  Кеширует на диск (TTL: часы/дни)
```

### Ключевые моменты

**Рендерит только для ботов** — обычные пользователи получают SPA как обычно.
Nginx различает по User-Agent: Googlebot, YandexBot, Twitterbot, TelegramBot, WhatsApp, facebookexternalhit...

**Кеширование** — страница рендерится один раз, результат сохраняется.
Повторные запросы ботов получают HTML из кеша. Типичный TTL: 24-72 часа.
Puppeteer запускается не на каждый запрос, а раз в сутки на страницу.

**Пре-прогрев** (опционально) — можно заранее скормить список URL (sitemap.xml),
и prerender-сервис обойдёт их все заранее, заполнив кеш до прихода ботов.

### Нагрузка на сервер

Для нашего каталога (~50 SKU, ~20 семейств):
- ~100 уникальных страниц
- Каждая рендерится ~3-5 секунд (Puppeteer)
- При кеше на 24 часа: ~100 рендеров в сутки
- Нагрузка: минимальная

Puppeteer потребляет ~200-300 MB RAM во время рендеринга, но работает эпизодически.

### Как развернуть

```bash
# 1. Установить
npm install prerender

# 2. Запустить сервис (порт 3000)
node server.js

# 3. Nginx: проксировать ботов
location / {
    # Проверка User-Agent
    if ($http_user_agent ~* "Googlebot|YandexBot|Twitterbot|TelegramBot|WhatsApp") {
        proxy_pass http://127.0.0.1:3000;  # prerender
    }
    # Обычные пользователи — SPA
    try_files $uri /index.html;
}
```

### Минусы

- Нужен Chrome/Chromium на сервере (~300 MB)
- Первый запрос бота к некешированной странице — задержка 3-5 сек
- Если контент изменился — бот увидит обновление только после протухания кеша

### Надёжность распознавания ботов

Риск не распознать бота минимальный. User-Agent поисковиков — публичная информация.

**Яндекс** (документация: webmaster.yandex.ru):
```
Mozilla/5.0 (compatible; YandexBot/3.0; +http://yandex.com/bots)
Mozilla/5.0 (compatible; YandexImages/3.0; +http://yandex.com/bots)
Mozilla/5.0 (compatible; YandexMedia/3.0; +http://yandex.com/bots)
```
Ключевое слово для фильтра: `YandexBot`. Яндекс прямо пишет в документации:
"Идентифицируйте нас по подстроке YandexBot".

**Google**:
```
Mozilla/5.0 (compatible; Googlebot/2.1; +http://www.google.com/bot.html)
Mozilla/5.0 AppleWebKit/537.36 (KHTML, like Gecko; compatible; Googlebot/2.1) Chrome/...
```
Ключевое слово: `Googlebot`

**Мессенджеры** (для OG-превью):
```
TelegramBot, WhatsApp, facebookexternalhit, Twitterbot, LinkedInBot, Slackbot
```

**Почему надёжно:**
- Поисковики не скрывают свой User-Agent — им выгодно чтобы сайты отдавали контент
- Prerender-сервисы используют эти списки годами — отработанная технология
- Если новый бот не попал в список — просто получит SPA как обычный пользователь, ничего не сломается

**Единственный риск:**
- Мобильная версия Googlebot может иметь нестандартный UA (но всё равно содержит "Googlebot")
- Экзотические боты (Bing, Baidu) могут не попасть в список — расширять по мере необходимости
- Prerender.io обновляет список автоматически; self-hosted нужно следить самому

**Вывод:** для Яндекса и Google — 100% надёжно, идентифицируются одинаково уже 10+ лет.
